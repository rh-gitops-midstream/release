# Backporting Bug Fixes for OpenShift GitOps

Bug fixes are made in the relevant component repository and then propagated to the appropriate OpenShift GitOps release branches. The exact process depends on the component.

## General Backport Process

1. Identify the affected OpenShift GitOps release versions.
2. Determine the component branch or version used by each affected release.
3. Implement or backport the fix in the appropriate component repository.
4. Ensure the change is reviewed, merged, and validated.
5. Update the component commit or version reference in the corresponding branch of the [`rh-gitops-midstream/release`](https://github.com/rh-gitops-midstream/release) repository.
6. Regenerate the source information.
7. Build and validate the affected OpenShift GitOps release artifacts.

## Component-specific Backport Instructions

### Standard Component Backport Process

The following components use the same general backport process:

* [Argo CD](https://github.com/argoproj/argo-cd)
* [Argo CD Agent](https://github.com/argoproj-labs/argocd-agent)
* [Argo Rollouts](https://github.com/argoproj/argo-rollouts)
* [Argo CD Image Updater](https://github.com/argoproj-labs/argocd-image-updater)
* [GitOps Console Plugin](https://github.com/redhat-developer/gitops-console-plugin)
* [GitOps Backend](https://github.com/redhat-developer/gitops-backend)

Follow these steps for each affected component:

1. Identify the affected OpenShift GitOps release versions.

2. Open the corresponding release branch in the `rh-gitops-midstream/release` repository.

3. Check the `sources` section of the branch’s `config.yaml` file to determine the component branch and commit used by that release.

4. Backport the bug fix to the identified branch in the component repository.

5. Ensure the fix is reviewed, merged, and validated in the component repository.

6. Update the component commit reference in the `sources` section of the affected release branch’s `config.yaml` file.

7. Run the following command in the release repository:

   ```bash
   make sources
   ```

8. Commit the updated `config.yaml` file and any other changes generated by the `make sources` target.

9. Create a pull request against the affected OpenShift GitOps release branch.

Repeat this process for every affected OpenShift GitOps release branch.

### Operator Bug Fixes

Operator-related bug fixes usually involve both the Argo CD Operator and the OpenShift GitOps Operator.

1. Identify the affected OpenShift GitOps release versions.

2. Check the affected release branch’s `config.yaml` file to determine the Argo CD Operator and GitOps Operator branches used by the release.

3. Implement or backport the fix to the appropriate branch of the `argocd-operator` repository.

4. Ensure the fix is reviewed, merged, and validated in `argocd-operator`.

5. Update the `argocd-operator` dependency in the corresponding `gitops-operator` branch by updating its `go.mod` file.

6. Run the required Go dependency commands, such as:

   ```bash
   go mod tidy
   ```

7. Review and commit the resulting `go.mod` and `go.sum` changes.

8. Ensure the dependency update is reviewed, merged, and validated in `gitops-operator`.

9. Update the GitOps Operator commit reference in the `sources` section of the affected release branch’s `config.yaml` file.

10. Run the following command in the release repository:

    ```bash
    make sources
    ```

11. Commit the updated `config.yaml` file and any other changes generated by the `make sources` target.

12. Create a pull request against the affected OpenShift GitOps release branch.

Repeat this process for every affected OpenShift GitOps release branch.

### Dex

OpenShift GitOps maintains a midstream Dex repository at [`rh-gitops-midstream/dex`](https://github.com/rh-gitops-midstream/dex).

The process differs depending on whether the change is a CVE fix or a regular bug fix.

#### CVE Fixes

CVE fixes may be backported directly to the appropriate release branch of the OpenShift GitOps Dex midstream repository.

When the fix is not already available in the main Dex repository, it should also be submitted upstream. Upstream may reject fixes for Dex versions that are no longer supported. Such a rejection does not block the required OpenShift GitOps midstream backport.

After the fix is merged into the midstream repository:

1. Update the Dex commit reference in the `sources` section of the affected release branch’s `config.yaml` file.

2. Run:

   ```bash
   make sources
   ```

3. Commit the generated changes and create a pull request against the affected OpenShift GitOps release branch.

4. Validate that the resulting Dex image contains the CVE fix.

#### Regular Bug Fixes

Regular bug fixes are generally not backported directly to the Dex midstream repository. Instead, the preferred approach is to update OpenShift GitOps to a newer Dex version that already contains the required fix.

The Dex version used by OpenShift GitOps is aligned with the Dex version used by the corresponding upstream Argo CD release. Therefore, a Dex version bump depends on the Dex version supported and used by upstream Argo CD.

Before updating Dex:

1. Check the Dex version used by the corresponding upstream Argo CD version.

2. Confirm that the required bug fix is included in that Dex version.

3. Update the Dex version and commit reference in the affected release branch’s `config.yaml` file.

4. Run:

   ```bash
   make sources
   ```

5. Commit the generated changes and create a pull request against the affected OpenShift GitOps release branch.

6. Validate the updated Dex version with the corresponding Argo CD version.

Refer to the README in the `rh-gitops-midstream/dex` repository for additional information about the Dex midstream maintenance and release process.

## Guidelines

* Backport only the commits required for the fix and any necessary dependencies.
* Avoid including unrelated changes.
* Resolve conflicts carefully and confirm that the resulting change preserves the original fix.
* Run component-specific tests before updating the release repository.
* Verify that the final images and release artifacts contain the expected fix.
* Record the component pull request, dependency-update pull request when applicable, release repository pull request, affected release streams, and validation results in the tracking issue.
